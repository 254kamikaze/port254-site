What is port254?

port254 is a comprehensive cybersecurity research platform designed to study and detect cyber threats targeting critical infrastructure systems and enterprise Active Directory environments. The platform combines industrial honeypots, Active Directory attack path analysis, and threat intelligence to improve security for both OT/ICS and IT systems.

Mission: We create realistic fake systems (honeypots) that cyber attackers think are real. When they try to hack them, we record everything to learn their tactics and develop better defenses for actual critical infrastructure and enterprise environments.

Why "port254"?

In industrial systems, "port" refers to communication channels (like doors) that systems use to talk to each other. Common industrial ports include 102 (S7), 502 (Modbus), and others. The "254" represents the Australian country code (+61) and Western Australia's area - this is a Perth-based security research project.

2. BloodHound Analysis Platform

A powerful Active Directory attack path analysis system that helps security teams understand and visualize attack paths in enterprise environments. Our implementation includes:

  • Multi-Engagement Support: Upload and manage multiple BloodHound datasets from different environments
  • 42 Preset Queries: Pre-built queries across 7 categories (Users, Computers, Groups, Permissions, Kerberos, Domain Trusts, Attack Paths)
  • Custom Cypher Queries: Write and execute custom Neo4j Cypher queries for advanced analysis
  • Attack Path Visualization: Discover and visualize attack paths between any two entities in Active Directory
  • HTTPS Upload: Secure upload of BloodHound ZIP files from anywhere on the internet
  • Data Isolation: Complete engagement-based filtering ensures data from different assessments never mix
  • Relationship Analysis: Full support for all BloodHound relationships including:
    • Delegation: AllowedToAct, AllowedToDelegate
    • Admin Rights: AdminTo, CanRDP, ExecuteDCOM, CanPSRemote
    • Permissions: WriteDacl, GenericAll, WriteOwner, AddKeyCredentialLink, ForceChangePassword
    • Group Membership: MemberOf
    • Domain Trusts: TrustedBy with full trust analysis

BloodHound Technology Stack

Our BloodHound implementation leverages:

  • Neo4j Graph Database: Single-database architecture with engagement-based isolation using engagement_id properties
  • Flask API Backend: RESTful API for upload, query, and management operations
  • Nginx Reverse Proxy: SSL termination, CORS handling, and load balancing
  • Docker Containers: Isolated services for API, Neo4j, and Nginx with custom networking
  • BloodHound Data Format: Full support for BloodHound v4 and v5+ JSON formats
  • Custom Upload Handler: Python-based parser with automatic relationship extraction for all ACE types

Architecture: Internet (GitHub Pages) → HTTPS (Port 5443) → Public IP (Port Forwarding) → Nginx Container (SSL/CORS) → Flask API Container → Neo4j Container → Graph Database

What We Learn

Every interaction with our platform provides valuable intelligence:

From Honeypots:

  • Attack Patterns: Which services do attackers target first? What commands do they run?
  • Credential Attacks: What username/password combinations are being tried?
  • Protocol Abuse: How are attackers manipulating industrial protocols like Modbus and S7?
  • Geographic Sources: Where in the world are attacks originating?
  • Tool Fingerprints: What hacking tools and scripts are being deployed?

From BloodHound Analysis:

  • Active Directory Weaknesses: Misconfigurations and privilege escalation paths
  • Domain Trust Relationships: Cross-domain attack vectors and SID filtering issues
  • Delegation Abuse: Unconstrained/constrained delegation and resource-based delegation risks
  • Kerberos Vulnerabilities: Kerberoasting, AS-REP roasting, and SPNs
  • Permission Chains: Complex ACE chains leading to Domain Admin compromise

Detection Development

The data collected feeds into our detection catalog - a library of security rules that can identify these attack patterns in real industrial and enterprise systems. Each detection is mapped to:

  • MITRE ATT&CK: Industry-standard framework for categorizing cyber threats
  • IEC 62443: International security standards for industrial automation
  • Real Telemetry: Actual data from our honeypot showing the detection in action

BloodHound Analysis Features

Query Categories

Our BloodHound implementation includes 42 preset queries organized into 7 categories:

  • Users (6 queries): All Users, Enabled Users, Disabled Users, Users Never Logged On, Users with Password Never Expires, Users with Reversible Encryption
  • Computers (4 queries): All Computers, Enabled Computers, Computers with Unconstrained Delegation, Computers with Constrained Delegation
  • Groups (5 queries): All Groups, High Value Groups, Domain Admins, Enterprise Admins, Groups with Most Members
  • Permissions (6 queries): Users with DCSync Rights, Users with GenericAll, Users with WriteOwner, Users with WriteDACL, Shortest Paths to Domain Admins, Shortest Paths to High Value Targets
  • Kerberos (4 queries): Kerberoastable Users, AS-REP Roastable Users, Users with SPN Set, Computers with LAPS Enabled
  • Domain Trusts (11 queries): All Domain Trusts, Bidirectional Trusts, One-Way Incoming Trusts, One-Way Outgoing Trusts, Forest Trusts, External Trusts, Trusts Without SID Filtering, Cross-Domain Admins, Foreign Security Principals, Cross-Trust Attack Paths, Domains in Environment
  • Attack Paths (6 queries): Shortest Path to Domain Admins, All Paths to Domain Admins, Shortest Path to High Value, Owned Principals Path to DA, Computers to DA (unconstrained delegation), Cross-Domain Privilege Escalation

Supported Relationships

Complete support for all BloodHound relationship types:

  • Delegation: AllowedToAct, AllowedToDelegate
  • Admin Rights: AdminTo, CanRDP, ExecuteDCOM, CanPSRemote
  • Ownership & Control: Owns, GenericAll
  • Permission Modification: WriteDacl, WriteOwner, GenericWrite
  • Credential Access: ForceChangePassword, ReadLAPSPassword, ReadGMSAPassword, AddKeyCredentialLink
  • DCSync: GetChanges, GetChangesAll, GetChangesInFilteredSet
  • Group Management: MemberOf, AddMember
  • Domain Relationships: TrustedBy with trust type, direction, and SID filtering analysis

Data Isolation: All queries automatically filter by selected engagement using engagement_id properties on all nodes and relationships, ensuring complete data isolation between different assessments.

System Architecture

BloodHound Multi-Engagement Architecture

Our BloodHound system uses a single Neo4j database with engagement-based isolation:

  • Single Database Design: All engagements share one Neo4j database instance for efficiency
  • Engagement ID Tagging: Every node and relationship has an engagement_id property
  • Automatic Query Filtering: All queries inject WHERE clauses to filter by selected engagement
  • Upload Processing: Custom Python handler parses BloodHound v4/v5+ ZIP files and extracts:
    • All node types (Users, Computers, Groups, Domains, GPOs, OUs, Containers)
    • All relationship types from various data sources (Aces, Members, Trusts, LocalAdmins, etc.)
    • Metadata and statistics for each engagement
  • Data Normalization: Consistent engagement_id format ensures proper filtering across uploads

Security Layers

  • Transport Security: All communication over HTTPS with SSL/TLS encryption
  • API Key Authentication: Required for all upload and query operations
  • CORS Configuration: Allows cross-origin requests for GitHub Pages integration
  • Query Safety: Only SELECT queries allowed (MATCH/RETURN), no write operations from frontend
  • Container Isolation: Each service runs in isolated Docker container with custom networking
  • Data Validation: Input sanitization and validation on all API endpoints

Safety & Ethics

No Real Systems at Risk

It's important to understand that nothing in the honeypot is real. There is no actual water treatment plant, no real pumps, no chemicals, and no public water supply connected to this system. Everything is simulated in a controlled environment.

Safety First: All honeypot systems are completely isolated from real infrastructure. Attackers cannot cause any real-world harm through our systems.

BloodHound Data Privacy

The BloodHound analysis platform is designed for authorized security assessments only:

  • Only upload BloodHound data from environments you have permission to assess
  • All uploaded data is stored securely with engagement-based isolation
  • Delete engagements when assessments are complete
  • Do not upload sensitive production data to internet-accessible instances

Responsible Disclosure

When we discover new attack techniques or vulnerabilities through our research, we work with industrial system vendors and security organizations to help protect real critical infrastructure. Our goal is to improve security for everyone, not to expose vulnerabilities maliciously.
