Export Detection Rule

Once you've tested and verified the detection works, export it in your SIEM's native format. One-click export means you don't have to manually translate the logic.

Export options:
‚Ä¢ Elastic (KQL): For Elasticsearch SIEM, Kibana
‚Ä¢ Splunk (SPL): For Splunk Enterprise Security
‚Ä¢ Sigma Rule: Universal format that converts to any SIEM
‚Ä¢ Microsoft Sentinel (KQL): For Azure Sentinel
‚Ä¢ QRadar (AQL): For IBM QRadar
Example Elastic KQL Export: event.category: "network" AND network.protocol: "s7" AND event.outcome: "failure" AND s7.function_code: "0x32" | where count() > 5 within 60s | alert severity: critical
üí° Next Step: Copy this rule, paste it into your SIEM's rule editor, and configure alert actions (email SOC, create ticket, etc.)

Deploy to Production SIEM

Take the exported rule and import it into your production SIEM. The exact steps vary by tool, but the concept is the same: create a new detection rule and paste the query.

Elastic SIEM Example:
1. Navigate to: Security ‚Üí Rules ‚Üí Create new rule
2. Select rule type: Custom query
3. Paste the KQL query from port254
4. Set severity to match detection (Critical/High/Medium/Low)
5. Add MITRE ATT&CK tags (shown in detection details)
6. Configure actions: email, webhook, create case
7. Enable the rule
‚ö†Ô∏è Tuning Required: Start with the rule as-is, then adjust thresholds based on your environment. Some networks are noisier than others. Monitor for false positives in the first week.
üí° Expected Alert Volume: Most OT detections fire 0-5 times per week in normal operations. If you're seeing 50+ alerts per day, you may need to tune thresholds.

Monitor & Respond

Now your detection is live. When it fires, you'll get real-time alerts about potential threats to your industrial systems. Use the port254 detection details to help with incident response.

When an alert fires:
1. Verify it's real: Check source IP, timing, affected system
2. Refer to port254: Review detection description for context
3. Check MITRE mapping: Understand the adversary technique
4. Contain if needed: Isolate affected PLC/SCADA system
5. Investigate root cause: Was it unauthorized access? Misconfiguration?
6. Document for compliance: Save evidence for auditors
üí° False Positive? If a legitimate process triggered the alert, add an exception in your SIEM for that specific source IP or process. Keep the detection enabled for everything else.
